Q4 - File Access and I/O Investigation
====================================

1) Creating a sample log file
Command:
echo "sample log line" > app.log
Output:
(no output)

2) Open files and file descriptors
Command:
exec 3<app.log
lsof -p $$ | grep app.log
Output:
bash 16066 codespace 3r REG 7,4 16 1310807 /workspaces/bits-exam/q4/app.log

Command:
ls -l /proc/$$/fd | grep app.log
Output:
lr-x------ 1 codespace codespace 64 Jul 3 17:22 3 -> /workspaces/bits-exam/q4/app.log

Observation:
The shell opened the file as file descriptor 3 for reading.

3) Standard output and error redirection
Command:
ls -l > stdout.txt
ls /not_a_real_path 2> stderr.txt
(ls -l && ls /not_a_real_path) > combined.txt 2>&1

Output:
(no output)

Resulting files:
stdout.txt
stderr.txt
combined.txt

Observation:
Standard output was redirected to stdout.txt, standard error was redirected to stderr.txt, and both streams were combined into combined.txt.

ulimit -a

Conclusion:
The shell has a large open-file limit and no practical file-size restriction, which is suitable for normal administration tasks.

5) Summary of findings
Linux manages file I/O through file descriptors, which are small integer handles used by processes. Redirection changes where standard output and standard error go, and resource limits define how many files or processes a shell can use.
